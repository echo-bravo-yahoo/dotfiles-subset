# Dotfiles Sync

Dotfiles are managed via `wildflower` in `~/workspace/dotfiles`. See that repo's `CLAUDE.md` for full details.

- `wildflower gather` — copies tracked files from the live filesystem into `meadows/`
- `wildflower sow` — copies files from `meadows/` back to the live filesystem

## Which files are tracked

Read `~/workspace/dotfiles/meadows.mjs` to determine tracked paths. Do not hardcode — the config is the source of truth.

## After editing a live dotfile

When a file matching a tracked path is edited on the live filesystem (e.g., `~/.claude/CLAUDE.md`, `~/.tmux.conf`, `~/.config/nvim/init.vim`), the dotfiles repo becomes stale. To sync:

```bash
cd ~/workspace/dotfiles && wildflower gather
```

**Only run `gather` when the intent is to commit the change.** The workflow is always:

1. `wildflower gather`
2. `git add` the specific changed files
3. `git commit`

Never gather without following through to a commit — an uncommitted gather leaves the repo dirty with unrelated changes.

## After editing a file in the dotfiles repo

When files under `~/workspace/dotfiles/meadows/` are edited directly, the live filesystem is stale. Follow the stash-gather-pop workflow to avoid clobbering external changes:

1. `git stash` — stash the working changes
2. `wildflower gather` — pull latest live state into `meadows/`
3. `git stash pop` — reapply changes on top of the fresh gather
4. Resolve any conflicts
5. `wildflower sow` — deploy the merged result to the live filesystem
6. Commit the final state

## When to skip

If an edit is explicitly temporary or experimental and the user says not to sync, skip the gather/sow step.

## Secrets (`~/.secrets.env`)

Secrets live in `~/.secrets.env`, generated by `sync-secrets.sh` in
the dotfiles repo. Key facts:

- **Cross-host.** The file is created on **every** host that runs
  `sync-secrets.sh` — macOS, most Linux, WSL. Contents are identical
  across hosts. There is no Mac-only "special" secrets file; the same
  1Password fetches populate the same env vars everywhere.
- **Staleness.** `sync-secrets.sh` only runs when invoked; if a new
  secret was added to the script (e.g., a new `export …` line), hosts
  that haven't re-run it won't have that var. Symptom: script errors
  like "API auth failed" / empty token. Fix: `bash
  ~/workspace/dotfiles/sync-secrets.sh` on the affected host.
- **Sourcing mechanism differs, not the file.** Most Mac/Linux hosts
  run zsh and pick `~/.secrets.env` up via `~/.zshenv`. Minimal
  embedded Linux hosts like `pi@stockholm` run bash and don't read
  `.zshenv` — so shells there don't auto-source secrets. This is a
  shell-config detail, not a platform one.
- **Scripts should source `~/.secrets.env` directly,** not
  `~/.zshenv`, so they work on any shell. The `~/.zshenv` path is
  silently broken on bash-only hosts.

```bash
# in a script that needs a secret:
[[ -f "$HOME/.secrets.env" ]] && source "$HOME/.secrets.env"
```

Adding a new secret: extend `sync-secrets.sh` with an
`export FOO="$(fetch foo_field)"`, add `foo_field` to the 1Password
"Dotfiles Secrets" item, then re-run `sync-secrets.sh` on every host
that needs it.
